# Accessibility Notes — Basira Design System

**Status:** Phase 2.5 substrate. Soft guard, not CI gate. Per Ahmad's
directive (`لا تعقّد`), this document and the helpers in `a11y-tokens.ts`
are advisory; the ministry's Digital Transformation team will run their
own WCAG audit before production rollout.

This file is the source of truth for a11y intent in Basira. It is read
together with the brand identity reference (`memory/reference_hrsd_brand_identity.md`)
and the dignity language constants (`design-system/dignity-language.ts`).

---

## 1. Why a11y matters here, specifically

Basira is a tool for services to people with disabilities. The audience
includes:

- **Beneficiaries** themselves — some with low vision, motor impairments,
  or cognitive load that affects their ability to read dense screens.
- **Staff** — long shifts (often 6-8 hours of screen time), some elderly
  social workers nearing retirement.
- **Family members** (future, via the family-bridge API) — generational
  range from young siblings to elderly parents.

Accessibility here is not a checklist. It is the same principle the
product itself rests on: **"محور الخدمة هو المستفيد"**. A tool that
fails an elderly social worker fails the beneficiary they serve.

## 2. The contrast hazard in HRSD's palette

The brand's secondary colors are designed for **headlines and accents**,
not body text. Three of them fail WCAG AA when used as foreground on
white:

| Color | Hex | Contrast on white | AA body? |
|---|---|---|---|
| Gold | `#FCB614` | ~1.6:1 | ❌ Fails |
| Orange | `#F7941D` | ~2.5:1 | ❌ Fails |
| Teal | `#269798` | ~4.6:1 | ✅ Borderline |
| Green | `#2BB574` | ~3.0:1 | ❌ Fails for body, OK for large text/UI |
| Cool Gray | `#7A7A7A` | ~4.5:1 | ✅ Borderline |
| Navy | `#0F3144` | ~14.5:1 | ✅ AAA |

The brand's own typography rule ("body text is gray or white only,
never secondary colors") happens to be the a11y-correct rule. Following
the brand book = passing AA for body text.

**Where you will hurt yourself**: status badges, callouts, banners. A
yellow badge with white text fails badly. Use `SAFE_PAIRS.badgeOnGold`
which puts navy text on gold instead — passes AAA.

## 3. Patterns to follow

### Body text

Always one of:
- `text-hrsd-cool-gray` (light mode body)
- `text-white` (dark mode body)
- `text-hrsd-navy` (headings only)

Never:
- `text-orange-*` / `text-yellow-*` / `text-amber-*` for body content
- Lightened tints (`text-teal-300`) on white

### Status badges

Use `SAFE_PAIRS` in `a11y-tokens.ts`:
```ts
import { SAFE_PAIRS } from '@/design-system/a11y-tokens';

<span style={{ color: SAFE_PAIRS.badgeOnGold.fg, backgroundColor: SAFE_PAIRS.badgeOnGold.bg }}>
  pending
</span>
```

### Focus rings

Use the canonical focus ring class from `FOCUS_RING.className`. It puts
a 2px gold ring offset 2px — visible against both navy headers and
white surfaces.

### Font scale

Tajawal is the production font. Default body is 14px. For surfaces
likely to be used by elderly staff (admissions intake, medication
review), the consuming component may opt into a larger base via
`text-base` or `text-lg` — never push body below 14px without an
explicit accessibility justification.

### RTL keyboard navigation

All interactive elements must be reachable via Tab in **logical reading
order** (right-to-left visually, but DOM order top-to-bottom). Don't
use absolute positioning to skip elements out of focus order.

## 4. Dignity overlap

Some a11y concerns are also dignity concerns. From
`dignity-language.ts`:

- `EmptyState` copy must describe what to do next, never use
  pity/deficit framing ("لا يوجد"). Screen readers announce empty
  states; what they announce matters.
- Status colors carry meaning beyond visual: "danger" + "incident"
  framing should follow `حدث يستدعي تعلّماً`, not `حادث`.
- Charts: the `<BrandPie>` clamp to 5 slices is partly an a11y rule —
  more than 5 slices reduces distinguishability for color-vision
  variations.

---

## 5. What this file deliberately omits

- jest-axe full suite — too noisy for a substrate phase, ministry will
  bring their own.
- Screen-reader transcripts — Arabic SR support varies across NVDA/JAWS;
  the ministry has internal QA tooling for this.
- ARIA region landmarks — each surface gets these as it's redesigned,
  not in a top-down sweep.

---

## 6. When you find a violation

1. If it's in a primitive (`Section`, `PageHeader`, etc.) — **fix the
   primitive**, the fix lands everywhere.
2. If it's in a one-off — fix locally and document the pattern here if
   it might recur.
3. If `warnIfFailsAA()` is firing in dev — that's the signal.

Soft guard, not a wall.
