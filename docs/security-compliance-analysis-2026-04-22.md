## 0. الخلاصة التنفيذية (قراءة سريعة — دقيقتان)

**الحالة العامة:** بصيرة في مرحلة تطويرية متقدمة. عدد من الضوابط الأساسية موجود (pgaudit، RLS، إخفاء الهوية الوطنية، TLS، بيئة .env مُحمية). لكن هناك **4 فجوات حرجة** لا مناص من معالجتها قبل التسليم، و **9 فجوات مهمة** يجب توثيقها للوكالة، و **6 عناصر** قابلة للتأجيل إلى ما بعد التسليم بإذن.

**القرارات المعمارية الحاسمة التي يجب إبلاغها للوكالة (لا قرار منّا):**

1. **منطقة استضافة Supabase:** سياسة `DT-IS-POL-400 §3.4.1` تنصّ على أنّ البيانات المُصنَّفة **يجب أن تُستضاف داخل المملكة**. Supabase حالياً لا تملك منطقة سعودية. هذا قرار وكالة — إمّا:
   (أ) الهجرة إلى مزوّد سعودي (SCCC، STC Cloud، Oracle KSA، Azure KSA)،
   (ب) الحصول على استثناء موثّق،
   (ج) إعادة التأطير كـ«بيانات غير حساسة» (غير منطقي لبيانات المستفيدين).
2. **MFA للحسابات ذات الصلاحيات:** لم تُفعَّل. Supabase يدعمها؛ الوكالة تُفعّلها عند النشر.
3. **ربط نفاذ:** نبني نقاط ربط (stubs) فقط — الوكالة تُكملها.
4. **سياسة الاحتفاظ بالبيانات:** مُعرَّفة في `Data Retention Policy v1.0` — تحتاج تطبيقاً تقنياً (PITR + سجلات التدقيق) على مستوى البنية التحتية، والوكالة تُقرّر مدّة الاحتفاظ الرسمية.

**ما يجب عمله الآن على `v2` (مسؤولية أحمد):**
- استبدال `SECURITY.md` القالبي بسياسة حقيقية مربوطة بلوائح HRSD.
- ترحيل 023 لإضافة تصنيفات البيانات (محدود/سري) كـ COMMENTS — كما فعلنا لأنواع العوق.
- ملف `docs/handover-security-packet.md` يُعدّ قائمة نقاط الربط للوكالة.
- تدوير كلمة سرّ قاعدة البيانات المسرّبة في `basira-diagnostic.mjs` (سبق التنبيه، لم يتم بعد).
- تفعيل Dependabot وإغلاق الثغرات الخمس المُعلَّمة على GitHub.
- فحص ذاتي للكود ضدّ تسريب أسرار (secrets scan).

**الإطار الزمني المُقترح:** 3 جلسات عمل (أسبوع كحد أقصى).

## 4. الفجوات المعمارية التي تتداخل بين Frontend / Backend / DB

**الفجوة 1 — التصنيف غير مُعلَن بصرياً:**
- **DB:** سنضيف COMMENT ON (المرحلة 1.5) — هذا طبقة بيانات.
- **Backend:** لا API يُعيد «مستوى التصنيف» مع البيانات.
- **Frontend:** لا شارة UI تُظهر حساسية البيانات.
- **الحل الكامل:** تأجيل التمظهر الواجهيّ للمرحلة 2. الآن DB فقط.

**الفجوة 2 — الجلسات والتوقيتات:**
- **DB:** Supabase يُدير الجلسة.
- **Backend:** API يستخدم bearer token.
- **Frontend:** `useAuthStore` يحفظ token.
- **الحل:** تعديل TTL في Supabase config (المرحلة 1.10). Frontend: إضافة حوار «انتهت الجلسة» عند HTTP 401.

**الفجوة 3 — pgaudit موجود لكن لا استعلام لنتائجه:**
- **DB:** تسجيل يُفعَّل، Postgres logs تحوي سطور AUDIT.
- **Backend:** لا endpoint لاستعلام.
- **Frontend:** لا لوحة إشرافية للمسؤول الأمنيّ.
- **الحل:** لوحة إشراف «سجلّات التدقيق» (مرحلة 3 — للوكالة).

**الفجوة 4 — المستخدم لا يَرى سياسة الخصوصية ولا يُوافق عليها:**
- **DB:** لا حقل `privacy_accepted_at` على المستخدم.
- **Backend:** لا فحص.
- **Frontend:** لا بوابة موافقة.
- **الحل:** صفحة + حقل + فحص (المرحلة 1.6).

**الفجوة 5 — سرّ DB مسرّب (basira-diagnostic.mjs):**
- **DB:** كلمة السرّ هذه قد تكون مفتاح service-role (صلاحيات كاملة).
- **Backend:** لا يستخدمها الإنتاج.
- **Frontend:** انتهى — أمر التجاهل في .gitignore ينفع للمستقبل، لا للماضي.
- **الحل:** تدوير SR key + تنظيف Git history (المرحلة 1.1 + 1.2).

## 6. الرسالة للقيادة عند التسليم (قالب)
