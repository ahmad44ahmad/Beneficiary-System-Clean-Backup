# تصنيف البيانات — Data Classification Reference

**الإصدار:** 1.0
**التاريخ:** 2026-04-22
**الإطار المرجعيّ:** `DT-IS-POL-1000 V10` (Data Classification Policy) + `Personal Data Protection Policy v1.0`
**المرجع التقنيّ:** `supabase/sql/023_data_classification.sql`
**المالك:** فريق بصيرة — يُحوَّل للوكالة عند التسليم

---

## 1. الغرض

هذه الوثيقة هي المرجع **النصّيّ المقروء** لمخطَّط تصنيف البيانات في نظام بصيرة. المرجع **التقنيّ المُلزِم** هو ملفّ ترحيل SQL رقم 023 الذي يُعلِّق على كلّ جدول وعمود حسّاس مباشرةً في قاعدة البيانات.

تخدم هذه الوثيقة:

1. مُراجعي الأمن السيبرانيّ الذين لا يقرؤون SQL.
2. فرق النشر والتدقيق في الوكالة.
3. صانعي القرار الذين يحتاجون فهم **ما الذي يُستَضاف** قبل اتّخاذ قرار الاستضافة أو الاحتفاظ.

---

## 2. مخطَّط التصنيف الوزاريّ

وفق `DT-IS-POL-1000 V10`، تُصنَّف البيانات إلى أربع مستويات:

| المستوى | الرمز | الوصف | مثال |
|---|---|---|---|
| **عامّ** | `Public` | بيانات يجوز نشرها دون ترخيص | رموز الإعاقة المعياريّة، فئات التقييم |
| **محدود** | `Restricted` | بيانات شخصيّة أو تشغيليّة حسّاسة مُتاحة لفئة مُحدَّدة | هويّة مستفيد، بيانات موظَّف، سجلّ يوميّ |
| **سرّيّ** | `Confidential` | بيانات قد يُؤثّر إفشاؤها على فرد أو مؤسّسة | تشخيص طبّيّ، بحث اجتماعيّ، حوادث |
| **سرّيّ للغاية** | `Top Secret` | بيانات استراتيجيّة وطنيّة | غير مُستخدَم في بصيرة |

**قاعدة عامّة في بصيرة:** أيّ بيان يعود لمستفيد معيَّن يُعامَل كـ«محدود» على الأقلّ. أيّ بيان طبّيّ أو اجتماعيّ تفصيليّ يعود له يُرفَع إلى «سرّيّ».

---

## 3. مصفوفة التصنيف حسب الجدول

تُلخِّص المصفوفة التالية كلّ الجداول الحسّاسة في بصيرة. للتفاصيل الكاملة والمبرِّرات انظر ملفّ الترحيل 023.

### 3.1 الجداول المُصنَّفة «عامّ»

| الجدول | السبب |
|---|---|
| `evaluation_criteria` | معايير تقييم مرجعيّة — لا بيانات شخصيّة |
| `benchmark_standards` | معايير أداء مرجعيّة |
| `grc_disability_codes` | رموز إداريّة معياريّة |
| جداول المرجعيّات (categories, units, lookup tables) | بنية ثابتة مشتركة |

### 3.2 الجداول المُصنَّفة «محدود»

| الجدول | الوصف المختصر | الأدوار المسموح لها بالقراءة |
|---|---|---|
| `beneficiaries` | البيانات الأساسيّة للمستفيدين | director, admin, doctor, social_worker, specialist, secretary |
| `employees` | بيانات الموظّفين الوظيفيّة | director, admin |
| `daily_care_logs` | السجلّات اليوميّة للرعاية | doctor, nurse, social_worker, director |
| `shift_handover_reports` | محاضر التسليم بين الفترات | staff, nurse, specialist, director |
| `fall_risk_assessments` | تقييم مخاطر السقوط | doctor, nurse, specialist |
| `risk_alerts` | تنبيهات مخاطر مُولَّدة آلياً | director, admin, quality role |
| `audit_logs` | سجلّ عمليّات النظام | admin فقط (read) — لا حذف |
| `strategic_kpis` | مؤشّرات الأداء الاستراتيجيّة | director, admin, quality |
| `beneficiary_classification` | تصنيف المستفيد حسب الاستقلاليّة | director, social_worker, specialist |
| `services_gap_analysis` | تحليل فجوات الخدمات | director, admin, quality |
| `independence_budget_analysis` | التحليل الماليّ للاستقلاليّة | director, admin |
| `human_rights_compliance` | الامتثال لحقوق الإنسان (CRPD) | director + جهة مُصرَّح لها بالعرض الخارجيّ |
| `external_services_benchmark` | مقارنات الخدمات الخارجيّة | director, admin, quality |
| `grc_risks` | سجلّ المخاطر المؤسسيّة | director, admin, quality |
| `catering_suppliers` | بيانات موردي الإعاشة | director, admin, catering manager |
| `contractor_evaluations` | تقييمات المقاولين | director, admin, quality |
| `iso_compliance_checklist` | قوائم الامتثال للمعايير الدوليّة | director, admin, quality |

### 3.3 الجداول المُصنَّفة «سرّيّ»

| الجدول | السبب |
|---|---|
| `fall_incidents` | محاضر حوادث السقوط — تفاصيل سلامة حرجة |
| `grc_safety_incidents` | حوادث السلامة — قد تُؤثّر على صورة المركز |
| `medical_profiles` (سيُعلَّق في v2 Phase 2) | الملفّ الطبّيّ الكامل |
| `medications` + `medication_administrations` (سيُعلَّق في v2 Phase 2) | الأدوية المصروفة — بيان طبّيّ حسّاس |
| `social_research` (سيُعلَّق بعد مراجعة الحقول) | البحث الاجتماعيّ — حسّاسيّة عاليّة جداً |

## 4. مصفوفة التصنيف على مستوى الأعمدة الحسّاسة

بعض الحقول داخل جداول «محدود» تستحقّ تصنيفاً أعلى بصفتها فرديّة:

| الجدول | العمود | التصنيف | ملاحظات |
|---|---|---|---|
| `beneficiaries` | `national_id` | محدود (معالجة خاصّة) | يُخفى في الواجهة إلى `***-***-NNNN`. القراءة الكاملة لـ director + admin + doctor + secretary فقط |
| `beneficiaries` | `medical_diagnosis` | سرّيّ داخل جدول محدود | سيُعاد تسميته إلى «التقييم الوظيفيّ» في v2 |
| `beneficiaries` | `guardian_phone` | محدود | لا يُعرض خارج نطاق المصرَّح له |
| `employees` | `phone` | محدود | لا يُعرض علنياً |
| `employees` | `email` | محدود | الاستخدام المؤسسيّ فقط |

## 5. قواعد المعالجة حسب التصنيف

### 5.1 «عامّ»

- يجوز العرض في أيّ واجهة.
- لا يُشترَط تسجيل القراءة.
- يجوز التصدير والنشر.

### 5.2 «محدود»

- العرض فقط للأدوار المُصرَّح لها (RLS).
- القراءة تُسجَّل عبر pgaudit إذا كانت على الجداول المُعرَّفة في `CLAUDE.md §3`.
- التصدير يتطلّب سجلّ عمليّة + تبرير.
- النقل عبر TLS فقط.

### 5.3 «سرّيّ»

- كلّ قراءة تُسجَّل — بدون استثناء.
- القائمة الكاملة للمصرَّح لهم مُحدَّدة في الجدول نفسه (RLS).
- لا تُصدَّر إلّا بإذن الإدارة العليا + بتبرير مكتوب.
- يُحظَر إرسالها إلى أيّ API خارجيّ (بما في ذلك خدمات AI — راجع `docs/ai-usage-policy.md`).
- الحفظ المحلّيّ على أجهزة المستخدمين ممنوع.

### 5.4 «سرّيّ للغاية»

- غير مُستخدَم في بصيرة. في حال أُضيف لاحقاً، يتطلّب سياسة مستقلَّة.

## 6. تطبيق التصنيف في الواجهة

التصنيف لا يظهر بصريّاً للمستخدم النهائيّ حالياً — هذا قرار واعٍ:

- **المُبرِّر:** بصيرة أداة تمكين لا أداة رقابة. إظهار شارات «سرّيّ» أمام كلّ سجلّ يُحوِّل التجربة إلى تجربة بيروقراطيّة.
- **البديل:** التصنيف يُطبَّق على طبقة قاعدة البيانات (RLS + pgaudit) بشكل غير مرئيّ. المستخدم يرى فقط ما يحقّ له رؤيته.
- **الاستثناء:** بعض واجهات التصدير (reports, export) تُعرَض فيها تذكرة «هذه البيانات مُحدَّدة الاستخدام» قبل الضغط على زرّ التصدير.

إذا طلبت الوكالة إظهار شارات التصنيف بصرياً، تُضاف طبقة UI مخصَّصة بعد النشر — مُخطَّط لها في v2 Phase 2.

## 7. مراجعة التصنيف

يُراجَع التصنيف في الحالات التالية:

1. **إضافة جدول جديد:** يجب تصنيفه فوراً عند ترحيله — خلاف ذلك يُعتبَر الترحيل ناقصاً.
2. **تغيير استخدام حقل:** إذا أُضيف للحقل غرض جديد قد يرفع حسّاسيّته.
3. **حادثة أمنيّة تتعلّق بالبيان:** قد يُرفَع التصنيف بناءً على الدرس.
4. **تحديث السياسة الوزاريّة:** عند إصدار نسخة جديدة من `POL-1000`.
5. **مراجعة سنويّة دوريّة.**

---

## 8. فجوات معروفة وتحفّظات

- بعض الجداول الحسّاسة لم تُعلَّق بعد في الترحيل 023 (مُوثَّقة في نهايته): `medications`, `medication_administrations`, `social_research`, `medical_profiles`. سيُكمَل ذلك في المرحلة اللغويّة من v2 بعد قرار إعادة التسمية من `patient` إلى `beneficiary` تقنياً.
- التصنيف حالياً على مستوى **الجدول + العمود**. لا يوجد تصنيف على مستوى **الصفّ**، أيّ أنّ سجلاً واحداً لا يُرفَع حسّاسيّته لأنّ محتواه غير اعتياديّ. هذا قرار واعٍ — الصفوف «الأكثر حسّاسيّة» تُحمى بـ RLS أكثر من تصنيفها.
- التصنيف لا يُطبَّق على بيانات `auth.users` لأنّ Supabase يُدير الجدول. المعالجة تتمّ إداريّاً عبر تقييد الوصول إلى لوحة Supabase.
